# 대학 중고거래 플랫폼 개발 기획서

이 문서는 대학 내 중고거래 플랫폼의 주요 기능별 데이터 흐름, 보안 요건 및 기술 스택을 정리한 기획서입니다.

---

## 1. 학교 인증 및 회원가입
- **사용자 주요 동작**: 학교 메일 인증, 인증번호 입력, ID/PW 설정, 닉네임 설정, 학번 입력
- **사용자 입력 데이터**: 학교 메일, 인증번호, ID/PW, 닉네임, 학번
- **시스템 생성 데이터**: 가입 승인일, 고유 UID, PW 해시값
- **보안**: PW 해시(Salt 적용), 학번/메일 등 개인식별정보(PII) 암호화
- **내부 DB 추천 & 이유**: **RDB (MySQL/PostgreSQL)** - 사용자 정보의 정합성과 트랜잭션 처리가 매우 중요함.
- **프런트엔드**: 이메일 형식 검증, 인증번호 타이머 구현, 회원가입 폼 데이터 유효성 체크
- **백엔드**: 인증메일 발송 로직, 비밀번호 암호화, JWT 토큰 발급 및 검증
- **사용 언어**: TS (Next.js) / Java (Spring Boot)
- **논의사항**: 졸업생 이용 여부, 닉네임 정책, 인증 주기

---

## 2. 내 정보 및 설정
- **사용자 주요 동작**: 프로필 사진 변경, 한 줄 소개 작성, 선호 지역 설정
- **사용자 입력 데이터**: 프로필 사진, 한 줄 소개, 선호 지역
- **시스템 생성 데이터**: 이미지 저장 URL, 접속 IP/기기 정보
- **보안**: 위치 정보 노출 제한 (정밀 좌표 대신 지역 ID 사용), 이미지 Metadata(EXIF) 제거
- **내부 DB 추천 & 이유**: **RDB + Object Storage (S3)** - 텍스트 정보는 RDB에, 무거운 이미지는 전용 저장소에 보관.
- **프런트엔드**: 이미지 파일 선택 및 미리보기, 이미지 리사이징(Client 측), 프로필 수정 API 호출
- **백엔드**: S3 업로드용 Presigned URL 생성, 유저 정보 DB 업데이트, 세션 관리
- **사용 언어**: TS (Next.js) / Node.js (NestJS)
- **논의사항**: 위치 정보 정밀도(좌표 or 장소), 개인정보 노출범위(학과, 학번), 알림 기본값

---

## 3. 판매 물건 등록/수정
- **사용자 주요 동작**: 상품명/카테고리 설정, 가격 입력, 상태 및 본문 작성, 사진 업로드
- **사용자 입력 데이터**: 상품명, 카테고리, 가격, 상태, 본문, 사진
- **시스템 생성 데이터**: 상품 고유 PID, 등록 시각, 판매 상태
- **보안**: 게시글 내 금칙어/스팸 필터링, 이미지 내 개인정보(폰번호 등) 자동 블러링
- **내부 DB 추천 & 이유**: **RDB + 검색 엔진 (Elasticsearch)** - 검색 성능 향상을 위해 텍스트는 검색 엔진에 인덱싱.
- **프런트엔드**: 카테고리 선택 UI, 다중 이미지 업로드 상태 관리, 등록 버튼 비활성화(중복 클릭 방지)
- **백엔드**: 이미지 메타데이터 제거, 상품 정보 DB 저장, 검색 엔진(Elasticsearch) 데이터 동기화
- **사용 언어**: TS (Next.js) / Java (Spring Boot)
- **논의사항**: 금지 품목 리스트, 가격 상한제

---

## 4. 탐색 및 상호작용
- **사용자 주요 동작**: 키워드 검색, 필터링, '찜' 클릭, 채팅 메시지 전송
- **사용자 입력 데이터**: 검색 키워드, 필터, '찜' 클릭, 채팅 메시지
- **시스템 생성 데이터**: 조회수, 찜 횟수, 채팅방 ID, 전송 시각
- **보안**: 채팅방 접근 제어(ACL), 메시지 내용 사생활 보호(종단간 암호화 고려)
- **내부 DB 추천 & 이유**: **NoSQL (MongoDB/Redis)** - 채팅은 쓰기 속도가 빠른 NoSQL, 찜/조회수는 Redis(캐시) 추천.
- **프런트엔드**: 무한 스크롤(Infinite Scroll) 구현, 실시간 채팅 UI, 소켓 연결 유지 및 메시지 렌더링
- **백엔드**: 웹소켓(WebSocket) 서버 운영, 메시지 DB 저장, 검색 쿼리 최적화(Indexing)
- **사용 언어**: Flutter / Node.js (Socket.io)
- **논의사항**: 채팅 기록 보관 기간, 검색 노출 알고리즘, 채팅 읽음 확인

---

## 5. 거래 완료 및 평가
- **사용자 주요 동작**: 거래 완료 확정, 별점 부여, 리뷰 작성, 매너 키워드 선택
- **사용자 입력 데이터**: 거래 완료 확정, 별점, 리뷰, 매너 키워드
- **시스템 생성 데이터**: 매너 온도 재계산, 거래 횟수 통계
- **보안**: 평점 조작 방지(어뷰징 체크), 거래 기록 위변조 방지
- **내부 DB 추천 & 이유**: **RDB (Transaction 필수)** - 결제 및 거래 완료 상태 변경은 데이터 충돌이 없어야 하므로 RDB 필수.
- **프런트엔드**: 거래 완료 모달 팝업, 별점 드래그 인터페이스, 평가 데이터 전송
- **백엔드**: 트랜잭션 처리(상태 변경+점수 합산), 신뢰도 점수 재계산 알고리즘 실행
- **사용 언어**: TS (Next.js) / Java (Spring Boot)
- **논의사항**: -

---

## 6. 불량 사용자 신고
- **사용자 주요 동작**: 신고 대상 지정, 사유 선택, 증거 사진 첨부
- **사용자 입력 데이터**: 신고 대상, 사유, 증거 사진
- **시스템 생성 데이터**: 신고 접수 번호, 누적 경고, 이용 제한 상태
- **보안**: 신고자 익명성 보장, 증거 자료(스크린샷) 무결성 및 접근 권한 제한
- **내부 DB 추천 & 이유**: **RDB** - 관리자 페이지와의 연동 및 법적 증거 자료로서의 기록 보존이 중요함.
- **프런트엔드**: 신고 사유 선택 팝업, 증거 사진 업로드 필드, 신고 완료 피드백 노출
- **백엔드**: 신고 내역 로그 저장, 관리자 알림(Slack/Mail) 발송, 누적 신고 시 계정 자동 일시 정지
- **사용 언어**: TS (Next.js) / Python (FastAPI)
- **논의사항**: 제재 수위(경고 누적제 or 바로 정지), 소명 절차, 블랙리스트 관리
